Architectural Approaches

Three ways to structure the Admin Command Center. All use Supabase RLS — the difference is how admin logic is organized on the Angular side.

A

Dedicated AdminService + Feature Module Recommended

New admin feature folder (src/app/features/admin/) with its own AdminService, components, and route. The existing supabase.ts stays untouched — admin SQL queries live exclusively in AdminService. A new adminGuard protects the route by checking the profiles table for the super_admin role.

Pros

  • Clean separation — admin code never bleeds into user-facing features
  • Matches existing pattern (host-dashboard has its own folder/service)
  • Easy to find, modify, or remove entirely
  • AdminService can call Supabase directly with RLS bypass policies

Cons

  • Some Supabase methods (e.g., sendMagicLink) may need to be duplicated or extracted
B

Extend Existing supabase.ts with Admin Methods

Add admin-specific methods (e.g., getAllEvents(), disableEvent(), sendManualMagicLink()) directly to the existing SupabaseService. No new service file — everything in one place.

Pros

  • Less files, simpler setup
  • Magic link method already lives in supabase.ts — reuse directly

Cons

  • supabase.ts already handles events, guests, RSVPs — adding admin bloats it further
  • Admin methods mixed with user methods — any component can accidentally call them
  • Harder to enforce "admin-only" at the service layer
C

Supabase Edge Functions for All Admin Mutations

Admin reads use RLS policies, but writes (disable event, delete event, edit fields) go through Supabase Edge Functions using the service role key server-side. Maximum security.

Pros

  • Service role key never touches the browser
  • Audit log at the Edge Function layer is possible

Cons

  • Significant complexity — each admin operation needs its own Edge Function
  • The supabase/functions/ folder is empty — this is a big greenfield addition
  • RLS with super_admin bypass policies already give server-enforced security without this overhead
  • Over-engineered for a single admin user

My recommendation: Approach A. It mirrors the existing host-dashboard pattern exactly — its own folder, its own service, its own guard. RLS policies on the Supabase side provide server-enforced security. The magic link method can be extracted to a shared util or called via the existing SupabaseService. Clean, proportionate, and maintainable.
