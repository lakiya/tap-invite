Design — Section 2: Auth, Routing & RLS

How the admin is identified, protected, and routed after login.

Auth Flow (updated)

auth-callback.component.ts — updated redirect logic
On SIGNED_IN event:
1. Fetch profiles row for current user
2. If role === 'super_admin' → navigate to /admin
3. Else → navigate to /dashboard

New adminGuard

src/app/core/guards/admin.guard.ts
1. Check supabase.getCurrentUser() — redirect /login if no session
2. Query profiles for role
3. If super_admin → allow
4. Else → redirect /dashboard (silently, no error page needed)

Regular users who somehow hit /admin just bounce to their own dashboard.

New ProfilesService

src/app/core/services/profiles/profiles.service.ts
getMyProfile(userId) → fetch role from profiles table
Used by adminGuard + auth-callback

Tiny service — one method. Keeps the Supabase client call out of the guard itself.

Updated Routes (app.routes.ts)

New route added
{ path: 'admin',
component: AdminDashboardComponent,
canActivate: [adminGuard] }

Supabase RLS Policies

New policies on events, guests, rsvps tables
SELECT all events → allowed if profiles.role = 'super_admin'
UPDATE any event → allowed if profiles.role = 'super_admin'
DELETE any event → allowed if profiles.role = 'super_admin'
SELECT all profiles → allowed if profiles.role = 'super_admin'
Existing host policies remain untouched. Cascade delete handles guests + rsvps automatically when event is deleted.
