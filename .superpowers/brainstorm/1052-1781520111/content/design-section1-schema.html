Design — Section 1: Database Schema

Two schema changes. Everything else in the admin panel is built on top of these.

1. New profiles table

profiles table
Column Type Notes
id uuid PK, references auth.users(id) ON DELETE CASCADE
email text Copied from auth.users for fast lookup
role user_role ENUM: 'user' | 'super_admin' — default: 'user'
created_at timestamptz default now()

A Postgres trigger on auth.users auto-inserts a row into profiles when a new user signs up (role = 'user'). To make yourself super_admin, run once in Supabase SQL editor:
UPDATE profiles SET role = 'super_admin' WHERE email = 'lakshikac@embla.asia';

2. Add is_enabled to events table

events table — new column
Column Type Notes
is_enabled boolean default true — only admin can set to false

3. Computed event status (UI-only, not stored)

Status logic derived at query time
Disabled → is_enabled = false (overrides date status)
Upcoming → is_enabled = true AND event_date > today
Ongoing → is_enabled = true AND event_date = today
Passed → is_enabled = true AND event_date < today

Computed in AdminService after fetching events. The verification guard fires on all four states — every admin edit requires typing "VERIFY".

Guest view fallback: The /w/:eventId/:guestId route already fetches event details. It will check is_enabled and render a clean fallback: "This invitation is temporarily unavailable. Please contact the host."
